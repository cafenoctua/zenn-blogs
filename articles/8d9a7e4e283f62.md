## 環境

- macOS
- Neovim: v0.11.5
- LazyVim: v15.13.0
- rust-analyzer: 0.3.2786（問題が発生したバージョン）

## 事象

Mac のストレージが 300GB 以上圧迫されていることに気づいた。「書類」カテゴリでの使用量が異常に大きく表示されており、原因がわからない状態だった。Neovim で Rust の開発をしていたが、まさかエディタがストレージを食い尽くしているとは思っていなかった。

まず `diskutil apfs list` でボリューム単位の使用量を確認した。

```bash
diskutil apfs list
```

ディスク構成：

- SSD 500GB
- Data ボリューム: 362GB 使用中（77.8%）

## 原因調査の流れ

### 1. ホームディレクトリの大まかな内訳を確認

```bash
du -sh ~/Documents ~/Downloads ~/Library ~/Pictures ~/Movies ~/Music
```

:::message
`df -h` はファイルシステム全体の空き容量を見るコマンドで、「どのディレクトリが容量を使っているか」はわからない。ディレクトリ単位で内訳を掘り下げるには `du` を使う。
:::

結果：Library が 25GB、その他は数GB 程度。合計しても 300GB に届かない。

### 2. 隠しフォルダに目を向ける

通常のフォルダには原因がなかったため、隠しフォルダ（ドットファイル）も含めて調べた。500MB 以上の隠しフォルダだけ GB 単位で表示するスクリプトを使う。

```bash
for d in /Users/<name>/.*; do
  size=$(du -sk "$d" 2>/dev/null | awk '{print $1}')
  if [ -n "$size" ] && [ "$size" -gt 500000 ]; then
    printf "%.2fG\t%s\n" "$(echo "$size / 1024 / 1024" | bc -l)" "$d"
  fi
done
```

**`~/.local` が 236GB** というとんでもない数字が出た。

### 3. .local の内訳を掘り下げる

```bash
du -sk ~/.local/* | sort -rn
# → .local/state  232GB
# → .local/share  3.75GB

du -sk ~/.local/state/* | sort -rn
# → .local/state/nvim  232GB

du -sk ~/.local/state/nvim/* | sort -rn
# → lsp.log  232GB  ← 犯人
```

`lsp.log` は Neovim の state ディレクトリに置かれるログファイルで、パスは `vim.fn.stdpath("state") .. "/lsp.log"` に相当する。`data` や `cache` ではなく `state` 以下なので、見落としやすい場所にある。

### 4. ログの中身を確認

```bash
tail -50 ~/.local/state/nvim/lsp.log
```

```
[ERROR][2026-02-28 09:50:09] ...p/_transport.lua:36  "rpc"  "rust-analyzer"  "stderr"
"WARN Propagating panic for cycle head that panicked in an earlier execution in that revision"
```

rust-analyzer が内部の依存グラフ解析（salsa というクエリエンジン）で循環を検出した際にパニックが伝播し、その警告が止まらなくなるというバグ。1行あたり約180バイトのログが同じ内容で延々と書き続けられており、**約13億行・232GB** に達していた。

## 原因の詳細

### 1. rust-analyzer の内部バグによるパニックループ

salsa（インクリメンタルコンパイルのクエリエンジン）の計算グラフ内でサイクルを検出した際、前回の実行でパニックした「サイクルの先頭ノード」のパニック状態が現在の実行に伝播するバグ。バージョン 0.3.2786 のリグレッションで、保存のたびに salsa のサイクル再計算がトリガーされ、そのたびにパニック伝播の警告が出続ける。

参考: [rust-analyzer Issue #21610](https://github.com/rust-lang/rust-analyzer/issues/21610), [rust-analyzer Issue #2446](https://github.com/rust-lang/rust-analyzer/issues/2446)

### 2. Neovim がデフォルトで WARN 以上のログを lsp.log に書き出す

デフォルトのログレベルは `WARN`。今回の rust-analyzer のエラーも WARN として stderr に出力されているため、デフォルト設定でそのまま記録される。

参照: [neovim/runtime/lua/vim/lsp/log.lua](https://github.com/neovim/neovim/blob/master/runtime/lua/vim/lsp/log.lua)

### 3. ログのローテーション機能がなく無制限に増加する

log.lua のソースコード上、1GB を超えると警告通知が出るが、ローテーション・自動削除は一切行われない。追記モードで書き続けるだけ。250GiB に達したユーザーの実例報告もある。

参照: [LunarVim Issue #4537](https://github.com/LunarVim/LunarVim/issues/4537), [Neovim Discourse: "LSP log file grows infinitely"](https://neovim.discourse.group/t/lsp-log-file-grows-infinitely/3596), [neovim Issue #32106](https://github.com/neovim/neovim/issues/32106)

## 対処

### 即時対応：ログファイルを削除

```bash
rm ~/.local/state/nvim/lsp.log
```

これだけで **232GB をそのまま回収**できた。

### 再発防止①：ログレベルを絞る

ログの書き出し量自体を減らすには、Neovim 起動時にログレベルを引き上げる。

```lua
-- lsp.log への出力を ERROR 以上に絞る（デフォルトは WARN）
vim.lsp.set_log_level("ERROR")

-- 完全に無効化する場合
vim.lsp.set_log_level("off")
```

ただしログを切ると LSP のトラブルシュート時に情報が取れなくなる。通常運用では `"ERROR"` に絞り、デバッグ時だけ `"WARN"` や `"DEBUG"` に戻す運用が現実的。

### 再発防止②：起動時に古いログを自動削除

`~/.config/nvim/lua/config/options.lua` に追加：

```lua
-- LSPログの肥大化防止: 起動時に1日以上古いlsp.logを削除
local lsp_log = vim.fn.stdpath("state") .. "/lsp.log"
local stat = vim.uv.fs_stat(lsp_log)
if stat and (os.time() - stat.mtime.sec) > 86400 then
  os.remove(lsp_log)
end
```

`vim.uv` は Neovim 0.10 以降で使える標準の libuv バインディング。
`stdpath("state")` は OS に依存せず正しいパスを返すので、移植性も高い。

### rust-analyzer 側の根本対処

バグ修正版へのアップデートと、salsa のキャッシュ状態リセットで解消できる。

```bash
# バグ修正版にアップデート
rustup update

# salsa のキャッシュ状態をリセット
cargo clean
```

## 参考: Neovim の stdpath ディレクトリ一覧

Neovim の `stdpath()` は OS ごとのパスを抽象化してくれる関数。`lsp.log` がどこにあるか迷ったときの参照用。

| 種別     | macOS / Linux のパス  | 用途                          |
| -------- | --------------------- | ----------------------------- |
| `config` | `~/.config/nvim`      | 設定ファイル（init.lua など） |
| `data`   | `~/.local/share/nvim` | プラグインデータ、shada など  |
| `state`  | `~/.local/state/nvim` | ログ、履歴など実行時の状態    |
| `cache`  | `~/.cache/nvim`       | 一時キャッシュ                |

## まとめ

- ストレージ圧迫の原因は通常フォルダだけでなく、**`du` で隠しフォルダも含めて調査する**
- `lsp.log` は `~/.local/state/nvim/` にある。肥大化していないか定期的に確認する
- Neovim の lsp.log にはローテーション機能がないため、**ログレベルを `"ERROR"` に絞るか、起動時自動削除の仕組みを入れておく**
- rust-analyzer のバグによるパニックループが起きたら、**`rustup update` と `cargo clean`** を試す
